WebSocket Gateway

pedidos.gateway.ts · NestJS + Socket.IO

El WebSocket Gateway es el corazón del sistema en tiempo real. Implementa dos interfaces de NestJS (OnGatewayConnection y OnGatewayDisconnect) para detectar cuando un dispositivo se conecta o desconecta, y expone múltiples manejadores de eventos mediante el decorador @SubscribeMessage. Cada evento que llega desde cualquier pantalla (cliente, camarero, cocina) se procesa aquí y se reemite al resto.

  • Cuando el cliente envía enviar_comanda, el servidor persiste el pedido en BD y lo reemite a la sala cocina y a todos los dispositivos.
  • Las salas de Socket.IO (unirse_sala) filtran qué eventos recibe cada pantalla, evitando ruido innecesario.
  • El evento solicitar_cuenta persiste el cambio en BD antes de emitirlo, garantizando consistencia si algún cliente se desconecta.
Evento Dirección Descripción
unirse_sala recibe Une al cliente a una sala lógica (cocina, mesa-X)
enviar_comanda R/E Persiste pedido y emite a cocina + broadcast
solicitar_cuenta R/E Persiste estado y notifica a todos los clientes
marcar_linea_kds R/E KDS marca plato como listo; puede emitir notificación sonora
pedido_actualizado emite Broadcast del pedido completo a todos los sockets

Autenticación JWT

auth.service.ts · auth.guard.ts · bcrypt + JWT

El sistema combina bcrypt para el hash de contraseñas y JWT para el control de sesiones sin estado. El flujo es: el usuario envía credenciales → se verifica la contraseña contra el hash en BD → se genera un token firmado con el secret del servidor. El token lleva embebidos el id, el correo y el rol del usuario, evitando consultas adicionales a BD en cada petición.

  • La opción «Recuérdame» cambia la expiración del token de 24h a 7d sin lógica adicional.
  • El AuthGuard extrae el token de la cabecera Authorization: Bearer, lo verifica y adjunta el payload a request['usuario'].
  • Los usuarios con activo: false son rechazados antes de verificar la contraseña.

Pedidos Service

pedidos.service.ts · Prisma ORM · Lógica de negocio

El PedidosService centraliza toda la lógica de negocio relacionada con pedidos y mesas. La pieza más crítica es crearOActualizarPedido: garantiza que cada mesa tenga como máximo un pedido activo durante una visita. Si ya existe uno en estado PENDIENTE, EN_COCINA, LISTO o ENTREGADO, las nuevas líneas se acumulan y el total se suma. Este diseño permite que el cliente pida varias rondas sin abrir cuentas nuevas.

  • Prisma permite crear líneas de pedido anidadas en la misma operación con lineas: { create: datos.lineas }, reduciendo las consultas a BD.
  • El método aplicarDescuentoAMesa inserta una línea con precio negativo, lo que mantiene el historial limpio sin alterar las líneas originales.
  • El KDS (Kitchen Display System) usa marcarLineaLista: cuando todas las líneas están marcadas, el pedido completo pasa a LISTO automáticamente.
  • cerrarCuenta acepta un desglose de pagos (efectivo, tarjeta, otro) que se persiste en el campo JSON pagos del pedido.

Mesas Service

mesas.service.ts · CRUD de mesas · Sesiones QR · Prisma

El MesasService gestiona todo el ciclo de vida de las mesas físicas del restaurante. Además del CRUD estándar, implementa la pieza más interesante del flujo de cliente: escanearQrYAbrirSesion, que convierte el escaneo de un QR en una sesión temporal con un token único. Este token viaja en la URL del cliente y vincula todos sus pedidos a la mesa correcta sin necesidad de autenticación.

  • crearMesa usa un doble paso: crea la mesa con qrUrl: '' para obtener el ID autogenerado de Prisma y en una segunda operación construye la URL real /qr/:id y la persiste.
  • La eliminación tiene una protección de estado: si la mesa no está INACTIVA lanza un BadRequestException, impidiendo borrar mesas con clientes sentados.
  • editarMesa valida unicidad de número excluyendo la propia mesa con id: { not: id }, evitando falsos conflictos al guardar sin cambiar número.
  • escanearQrYAbrirSesion es idempotente: si la mesa ya tiene sesión activa (cliente recargó o escaneó dos veces), devuelve el token existente en lugar de crear uno nuevo.

Socket Service

socket.service.ts · Angular + Socket.IO + NgZone

El SocketService encapsula toda la comunicación WebSocket del frontend. La clave del diseño es el uso de NgZone.run(): cuando Socket.IO emite un evento, el callback ocurre fuera de la zona de detección de cambios de Angular, lo que impide que la UI se actualice automáticamente. Envolver cada callback en zone.run() garantiza que Angular detecte el cambio y re-renderice el componente.

  • El método escuchar() devuelve un Observable de RxJS, lo que permite suscribirse y combinar eventos con operadores reactivos.
  • emitirComanda() usa el patrón de acknowledgment de Socket.IO para esperar la confirmación del servidor antes de resolver la Promise.
  • El servicio se registra con providedIn: 'root', garantizando una única instancia compartida por toda la aplicación.
socket.service.ts
TypeScript

2export class SocketService {

4  private zone = inject(NgZone);


5


6  constructor() {

8  }

16    });


17  }

22        this.zone.run(() => {


23          if (respuesta.status === 'ok') resolve(respuesta.pedido);


24          else reject(respuesta.mensaje);


25        });


26      });


27    });


28  }


29}

Guards & Routing

app.routes.ts · authGuard · rolGuard

Angular protege las rutas con dos functional guards: authGuard comprueba que exista un token válido en el almacenamiento, y rolGuard verifica que el rol del usuario esté incluido en el array roles definido en route.data. Ambos guards se componen en el array canActivate de cada ruta, ejecutándose en orden.

  • Las rutas públicas (/qr/:id, /menu) no tienen guards y son accesibles sin autenticación.
  • El rolGuard usa ModalNotificacionService para mostrar un mensaje de error antes de redirigir, en lugar de fallar silenciosamente.
  • El layout del dashboard (DashboardLayout) actúa como ruta padre que aplica authGuard a todas sus rutas hijo de una sola vez.

Auth Interceptor

auth.interceptor.ts · HttpInterceptorFn

El authInterceptor intercepta todas las peticiones HTTP de Angular y les añade automáticamente la cabecera Authorization: Bearer <token>. Además, escucha las respuestas 401 Unauthorized del servidor para cerrar la sesión y redirigir al login cuando el token haya expirado, excepto en la zona pública del cliente (/menu).

  • Usa el patrón clone: las peticiones HTTP de Angular son inmutables, por lo que se crea una copia modificada (req.clone()) con la cabecera añadida.
  • El operador catchError de RxJS captura errores HTTP y reacciona a ellos sin romper el flujo de la aplicación.
auth.interceptor.ts
TypeScript
1export const authInterceptor: HttpInterceptorFn = (req, next) => {


2  const authService = inject(AuthService);


3  const router     = inject(Router);


4  const token      = authService.obtenerToken();


5


6  const peticionClonada = token


7    ? req.clone({ setHeaders: { Authorization: `Bearer ${token}` } })


8    : req;


9


10  return next(peticionClonada).pipe(


11    catchError((error: HttpErrorResponse) => {


12      const esZonaCliente = router.url.includes('/menu');


13      if (error.status === 401 && !esZonaCliente) {


14        authService.logout();


15        router.navigate(['/login']);


16      }


17      return throwError(() => error);


18    })


19  );


20};

Mesas Component

mesas.ts · Angular 17 Standalone · Mini TPV integrado

El componente Mesas es el más complejo del frontend. Combina la gestión visual de las mesas del restaurante con un mini TPV integrado que permite a los camareros abrir la carta, añadir productos al ticket y enviar comandas directamente desde el dashboard. La estrategia de sincronización es conservadora: ante cualquier evento WebSocket de cambio, el componente recarga los datos del servidor en lugar de fusionar estado local, eliminando posibles desincronizaciones.

  • El ngOnInit se suscribe a cuatro eventos WebSocket simultáneos: pedido_actualizado, mesas_actualizadas, cuenta_cerrada y cuenta_solicitada.
  • Usa el patrón moderno de Angular 17+ con inject() en lugar del constructor para inyectar los servicios (MesaService, CartaService, SocketService, AuthService, TranslateService).
  • El modal de cobro implementa un mini desglose de pagos (efectivo, tarjeta, otro) con cálculo de cambio y cobertura mínima antes de confirmar.
  • El sistema de descuentos por porcentaje o importe fijo se aplica emitiendo un evento Socket, delegando la lógica al backend.
  • imprimirPreTicket usa window.print() con un timeout para capturar la hora exacta antes de lanzar el diálogo del sistema.

Carta Component

carta.ts · Angular 17 Standalone · CRUD de menú + i18n

El componente Carta es el panel de administración del menú del restaurante. Permite gestionar categorías (crear, editar, eliminar con validación) y productos (crear, editar, toggle de disponibilidad, eliminar), todo con soporte bilingüe es/en. Cada mutación emite el evento notificar_cambio_carta por WebSocket para que el menú del cliente (/menu) se actualice en tiempo real sin necesidad de recargar la página.

  • El getter computado resultadosBusqueda realiza una búsqueda cross-categoría filtrando por nombre_es, nombre_en, descripcion_es y descripcion_en simultáneamente.
  • La eliminación de categoría tiene una guarda previa: si la categoría contiene productos, bloquea la acción y avisa al usuario antes de llegar al servidor.
  • toggleEstado actualiza el flag disponible del producto en local inmediatamente (optimistic update) y luego persiste en BD con CartaService.
  • Tras cualquier operación de escritura, cargarCarta recarga la lista y actualiza la referencia de categoriaSeleccionada con los nuevos datos sin resetear la selección del usuario.
