WebSocket Gateway
back/src/pedidos/pedidos.gateway.ts

El WebSocket Gateway es el corazón del sistema en tiempo real. Implementa dos interfaces de NestJS (OnGatewayConnection y OnGatewayDisconnect) para detectar cuando un dispositivo se conecta o desconecta, y expone múltiples manejadores de eventos mediante el decorador @SubscribeMessage. Cada evento que llega desde cualquier pantalla (cliente, camarero, cocina) se procesa aquí y se reemite al resto.

  • Cuando el cliente envía enviar_comanda, el servidor persiste el pedido en BD y lo reemite a la sala cocina y a todos los dispositivos.
  • Las salas de Socket.IO (unirse_sala) filtran qué eventos recibe cada pantalla, evitando ruido innecesario.
  • El evento solicitar_cuenta persiste el cambio en BD antes de emitirlo, garantizando consistencia si algún cliente se desconecta.
pedidos.gateway.ts — Clase principal y conexiones
BACKEND 
@WebSocketGateway({ cors: { origin: '*' } })
export class PedidosGateway implements OnGatewayConnection, OnGatewayDisconnect {
  @WebSocketServer() server: Server;

  handleConnection(client: Socket) {
    console.log(`🟢 Cliente conectado: ${client.id}`);
  }

  @SubscribeMessage('unirse_sala')
  handleUnirseSala(
    @ConnectedSocket() client: Socket,
    @MessageBody() sala: string
  ) {
    client.join(sala); // Une al cliente a una sala (ej: 'cocina', 'mesa-1')
    return { evento: 'sala_unida', mensaje: `Unido a ${sala}` };
  }
}
pedidos.gateway.ts — Envío de comanda en tiempo real
BACKEND 
@SubscribeMessage('enviar_comanda')
async handleEnviarComanda(
  @ConnectedSocket() client: Socket,
  @MessageBody() payload: any
) {
  try {
    // 1. Persistir en base de datos (crea o acumula al pedido activo)
    const pedidoActivo = await this.pedidosService
      .crearOActualizarPedido(payload);

    // 2. Emitir solo a la sala de cocina
    this.server.to('cocina').emit('nuevo_pedido_cocina', pedidoActivo);

    // 3. Notificar a todas las pantallas del cambio de estado
    this.server.emit('pedido_actualizado', pedidoActivo);
    this.server.emit('mesas_actualizadas');

    return { status: 'ok', pedido: pedidoActivo };
  } catch (error) {
    return { status: 'error', mensaje: 'No se pudo procesar el pedido' };
  }
}

@SubscribeMessage('solicitar_cuenta')
async handleSolicitarCuenta(@MessageBody() mesaId: string) {
  // Persistir en BD antes de emitir: garantiza consistencia ante desconexiones
  await this.prisma.mesa.update({
    where: { id: mesaId },
    data:  { cuentaSolicitada: true }
  });
  const mesa = await this.prisma.mesa.findUnique({
    where:  { id: mesaId },
    select: { numero: true }
  });
  this.server.emit('cuenta_solicitada', { mesaId, numero: mesa?.numero });
}

@SubscribeMessage('marcar_linea_kds')
async handleMarcarLinea(
  @MessageBody() payload: { pedidoId: string; lineaId: string; listo: boolean }
) {
  const pedidoActualizado = await this.pedidosService
    .marcarLineaLista(payload.pedidoId, payload.lineaId, payload.listo);
  this.server.emit('pedido_actualizado', pedidoActualizado);
  // Notificación sonora en el terminal cuando el plato está listo
  if (payload.listo && pedidoActualizado) {
    this.server.emit('plato_listo_notificacion', pedidoActualizado.mesaId);
  }
}

Autenticación JWT + bcrypt
back/src/auth/auth.service.ts & auth.guard.ts

El sistema de autenticación combina bcrypt para el hash de contraseñas y JWT para el control de sesiones sin estado. El flujo es: el usuario envía credenciales → se verifica la contraseña contra el hash en BD → se genera un token firmado con el secret del servidor. El token lleva embebidos el id, el correo y el rol del usuario, evitando consultas adicionales a BD en cada petición.

  • La opción «Recuérdame» cambia la expiración del token de 24h a 7 días sin lógica adicional.
  • El AuthGuard extrae el token de la cabecera Authorization: Bearer, lo verifica y adjunta el payload a request['usuario'] para que los controladores conozcan al usuario sin consultar BD.
  • Los usuarios desactivados (activo: false) son rechazados antes de verificar la contraseña.
auth.service.ts
BACKEND 
async login(
  correo: string,
  contrasenaPlana: string,
  recuerdame: boolean = false
) {
  const usuario = await this.prisma.usuario
    .findUnique({ where: { correo } });

  if (!usuario)
    throw new UnauthorizedException('Credenciales incorrectas');

  if (!usuario.activo)
    throw new UnauthorizedException('Usuario desactivado');

  const ok = await bcrypt.compare(
    contrasenaPlana, usuario.contrasena
  );
  if (!ok)
    throw new UnauthorizedException('Credenciales incorrectas');

  const payload = {
    sub:    usuario.id,
    correo: usuario.correo,
    rol:    usuario.rol
  };
  const expiresIn = recuerdame ? '7d' : '24h';

  return {
    access_token: await this.jwtService
      .signAsync(payload, { expiresIn }),
    usuario: { id: usuario.id, nombre: usuario.nombre, rol: usuario.rol }
  };
}
auth.guard.ts
BACKEND 
@Injectable()
export class AuthGuard implements CanActivate {

  async canActivate(
    context: ExecutionContext
  ): Promise<boolean> {
    const request = context
      .switchToHttp().getRequest();
    const token =
      this.extraerTokenDeCabecera(request);

    if (!token)
      throw new UnauthorizedException();

    try {
      const payload = await this.jwtService
        .verifyAsync(token, {
          secret: process.env.JWT_SECRET
        });
      // El payload (id, correo, rol) queda
      // disponible en todos los controladores
      request['usuario'] = payload;
    } catch {
      throw new UnauthorizedException(
        'Token inválido o expirado'
      );
    }
    return true;
  }

  private extraerTokenDeCabecera(
    request: Request
  ) {
    // Authorization: Bearer <token>
    const [tipo, token] =
      request.headers.authorization
        ?.split(' ') ?? [];
    return tipo === 'Bearer'
      ? token : undefined;
  }
}

Lógica de negocio con Prisma ORM
back/src/pedidos/pedidos.service.ts

El método crearOActualizarPedido implementa la lógica clave del sistema: una mesa siempre tiene como máximo un pedido activo. Si ya existe uno (en estado PENDIENTE, EN_COCINA, LISTO o ENTREGADO), las nuevas líneas se acumulan sobre él y el total se suma. Si no hay pedido activo, se crea uno nuevo. Este diseño permite que el cliente pida varias veces durante la misma visita sin abrir cuentas nuevas.

  • Prisma permite crear líneas de pedido anidadas en la misma operación con lineas: { create: datos.lineas }, reduciendo las consultas a BD.
  • El include con relaciones anidadas devuelve el pedido completo en una sola query, listo para emitir vía WebSocket.
  • La mesa se actualiza a estado ACTIVA automáticamente al crear el primer pedido.
pedidos.service.ts — crearOActualizarPedido()
BACKEND 
async crearOActualizarPedido(datos: {
  mesaId: string;
  sesionId?: string;
  lineas: { productoId: string; cantidad: number;
            notas?: string; precioUnitario: number }[]
}) {
  const totalNuevo = datos.lineas.reduce((acc, l) =>
    acc + (l.cantidad * l.precioUnitario), 0);

  // ¿Hay ya una cuenta abierta en esta mesa?
  let pedido = await this.prisma.pedido.findFirst({
    where: {
      mesaId: datos.mesaId,
      estado: { in: ['PENDIENTE','EN_COCINA','LISTO','ENTREGADO'] }
    }
  });

  if (pedido) {
    // SÍ hay cuenta: acumulamos líneas y sumamos al total
    pedido = await this.prisma.pedido.update({
      where: { id: pedido.id },
      data: {
        estado: 'PENDIENTE',
        total: Number(pedido.total) + totalNuevo,
        lineas: { create: datos.lineas }
      },
      include: {
        lineas: { include: { producto: true } },
        mesa: true
      }
    });
  } else {
    // NO hay cuenta: nuevo pedido
    pedido = await this.prisma.pedido.create({
      data: {
        mesaId:  datos.mesaId,
        sesionId: datos.sesionId,
        total:   totalNuevo,
        estado:  'PENDIENTE',
        lineas:  { create: datos.lineas }
      },
      include: {
        lineas: { include: { producto: true } },
        mesa: true
      }
    });
  }

  // La mesa pasa a estado ACTIVA automáticamente
  await this.prisma.mesa.update({
    where: { id: datos.mesaId },
    data:  { estado: 'ACTIVA' }
  });
  return pedido;
}

Socket Service — Angular
front/src/app/core/services/socket.ts

El SocketService encapsula toda la comunicación WebSocket del frontend. La clave del diseño es el uso de NgZone.run(): cuando Socket.IO emite un evento, el callback ocurre fuera de la zona de Angular (zona de detección de cambios), lo que impide que la UI se actualice automáticamente. Envolver cada callback en zone.run() garantiza que Angular detecte el cambio y re-renderice el componente.

  • El método escuchar() devuelve un Observable de RxJS, lo que permite a los componentes suscribirse y combinar eventos con operadores reactivos.
  • emitirComanda() usa una Promise con el patrón de acknowledgment de Socket.IO para esperar la confirmación del servidor antes de resolver.
  • El servicio se registra con providedIn: 'root', por lo que existe una única instancia compartida por toda la aplicación.
socket.ts
FRONTEND 
@Injectable({ providedIn: 'root' })
export class SocketService {
  private socket: Socket;
  private zone = inject(NgZone);

  constructor() {
    this.socket = io('http://localhost:3000', {
      transports: ['websocket'],
      autoConnect: true
    });
  }

  // Devuelve un Observable que Angular puede consumir reactivamente
  escuchar(nombreEvento: string): Observable<any> {
    return new Observable((observer) => {
      this.socket.on(nombreEvento, (data) => {
        // NgZone: fuerza a Angular a detectar el cambio
        this.zone.run(() => observer.next(data));
      });
      // Cleanup: se desuscribe al destruir el componente
      return () => this.socket.off(nombreEvento);
    });
  }

  // Acknowledgment pattern: espera confirmación del servidor
  emitirComanda(datosPedido: any): Promise<any> {
    return new Promise((resolve, reject) => {
      this.socket.emit('enviar_comanda',
        datosPedido, (respuesta: any) => {
          this.zone.run(() => {
            if (respuesta.status === 'ok')
              resolve(respuesta.pedido);
            else
              reject(respuesta.mensaje);
          });
        });
    });
  }

  emitir(nombreEvento: string, datos: any) {
    this.socket.emit(nombreEvento, datos);
  }

  unirseASala(nombreSala: string) {
    this.socket.emit('unirse_sala', nombreSala);
  }
}

Guards de ruta y control de roles
front/src/app/core/guards/ & app.routes.ts

Angular protege las rutas con dos functional guards: authGuard comprueba que exista un token válido en el almacenamiento, y rolGuard verifica que el rol del usuario esté incluido en el array roles definido en route.data. Ambos guards pueden componerse en el array canActivate de cada ruta, ejecutándose en orden.

  • Las rutas públicas (/qr/:id, /menu) no tienen guards y son accesibles sin autenticación.
  • El rolGuard usa el ModalNotificacionService para mostrar un mensaje de error antes de redirigir, en lugar de fallar silenciosamente.
  • El layout del dashboard (DashboardLayout) actúa como ruta padre que aplica authGuard a todas sus rutas hijo de una sola vez.
auth-guard.ts + rol-guard.ts
FRONTEND 
// authGuard — requiere token válido
export const authGuard: CanActivateFn =
  (route, state) => {
    const auth   = inject(AuthService);
    const router = inject(Router);

    if (auth.obtenerToken()) return true;

    router.navigate(['/login']);
    return false;
  };

// rolGuard — verifica rol del usuario
export const rolGuard: CanActivateFn =
  (route, state) => {
    const auth    = inject(AuthService);
    const router  = inject(Router);
    const modal   = inject(ModalNotificacionService);

    const roles   = route.data['roles'] as string[];
    const usuario = auth.obtenerUsuarioActual();

    if (usuario && roles.includes(usuario.rol))
      return true;

    modal.mostrarError(
      'Tu rol no tiene permisos para esta sección.'
    );
    router.navigate(['/dashboard']);
    return false;
  };

Interceptor HTTP — Inyección automática del token
front/src/app/core/guards/auth.interceptor.ts

El authInterceptor intercepta todas las peticiones HTTP que Angular realiza y les añade automáticamente la cabecera Authorization: Bearer <token>. Esto evita tener que añadir el token manualmente en cada llamada a la API. Además, escucha las respuestas 401 Unauthorized del servidor para cerrar la sesión y redirigir al login cuando el token haya expirado, excepto en la zona pública del cliente (/menu).

  • Usa el patrón clone: las peticiones HTTP de Angular son inmutables, por lo que se crea una copia modificada (req.clone()) con la cabecera añadida.
  • El operador catchError de RxJS permite capturar errores HTTP y reaccionar a ellos sin romper el flujo de la aplicación.
auth.interceptor.ts
FRONTEND 
export const authInterceptor: HttpInterceptorFn =
  (req, next) => {
    const authService = inject(AuthService);
    const router      = inject(Router);

    const token = authService.obtenerToken();

    // Clonar la petición y añadir la cabecera
    const peticionClonada = token
      ? req.clone({
          setHeaders: {
            Authorization: `Bearer ${token}`
          }
        })
      : req;

    return next(peticionClonada).pipe(
      catchError((error: HttpErrorResponse) => {
        const esZonaCliente =
          router.url.includes('/menu');

        // 401 fuera del área pública → logout + login
        if (error.status === 401 && !esZonaCliente) {
          authService.logout();
          router.navigate(['/login']);
        }
        return throwError(() => error);
      })
    );
  };

Componente Mesas — Tiempo real en Angular
front/src/app/features/dashboard/mesas/mesas.ts

El componente Mesas es el más complejo del frontend. En su ngOnInit se suscribe a múltiples eventos WebSocket simultáneamente para mantener la vista siempre sincronizada. La estrategia elegida para manejar la consistencia es conservadora: ante cualquier evento de cambio, en lugar de intentar fusionar el estado local con el dato nuevo, el componente recarga directamente los datos del servidor, garantizando que nunca haya desincronizaciones.

  • La suscripción a pedido_actualizado actualiza la mesa afectada en el array local sin recargar todo.
  • La suscripción a mesas_actualizadas recarga la lista completa de mesas desde el servidor.
  • El componente usa inyección de dependencias con inject() en lugar del constructor, patrón moderno de Angular 14+.
mesas.ts — ngOnInit y suscripciones WebSocket
FRONTEND 
@Component({
  selector: 'app-mesas',
  standalone: true,
  imports: [CommonModule, QRCodeComponent, FormsModule, TranslateModule],
  templateUrl: './mesas.html',
})
export class Mesas implements OnInit {
  private socketService = inject(SocketService);
  private mesaService   = inject(MesaService);
  // ... otros servicios inyectados

  mesas: any[] = [];
  rolUsuario: string = '';

  ngOnInit() {
    this.cargarMesas();

    // Cuando cambia un pedido: actualizar esa mesa en local
    this.socketService
      .escuchar('pedido_actualizado')
      .subscribe((pedido: any) => {
        if (!pedido) return;
        const i = this.mesas
          .findIndex(m => m.id === pedido.mesaId);
        if (i !== -1)
          this.mesas[i].pedidoActivo = pedido;
      });

    // Cuando cambia el estado de la sala: recargar todo
    this.socketService
      .escuchar('mesas_actualizadas')
      .subscribe(() => this.cargarMesas());

    // Cuando se cierra una cuenta: limpiar estado local
    this.socketService
      .escuchar('cuenta_cerrada')
      .subscribe((mesaId: string) => {
        const mesa = this.mesas
          .find(m => m.id === mesaId);
        if (mesa) mesa.pedidoActivo = null;
        this.cargarMesas();
      });

    // Cuando el cliente solicita la cuenta: activar aviso visual
    this.socketService
      .escuchar('cuenta_solicitada')
      .subscribe((payload: any) => {
        const mesa = this.mesas
          .find(m => m.id === payload.mesaId);
        if (mesa)
          mesa.cuentaSolicitada = true;
      });
  }
}
